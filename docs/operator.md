# Operator Notes

## Current state

A separate controller/operator Deployment is not implemented yet.
The current PoC is agent-driven:

- Desired state lives in `PowerPolicy` CRDs.
- Each node agent reads all policies and self-selects the one matching its node labels.

This keeps the implementation small while preserving a clean path to a centralized operator.

## Planned operator responsibilities

A future operator should:

- Build per-node desired power actions from telemetry and policy logic.
- Track inventory/capabilities by node (CPU/GPU support).
- Validate policy conflicts and publish status/conditions.
- Optionally expose `status` on node-scoped objects.

## Migration path (minimal disruption)

To evolve from current mode to full operator mode:

1. Keep `PowerPolicy` as user-facing intent.
2. Add a node-scoped CRD (for example `NodePowerState`) generated by operator.
3. Switch agent from "read all policies" to "read local node state".
4. Keep backend plugin interface in agent for Intel/GPU additions.

## Suggested operator deployment shape

- Deployment in `joulie-system`
- ServiceAccount + RBAC with read nodes, read/write relevant CRDs
- Optional leader election
- Optional metrics endpoint
